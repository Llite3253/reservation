# 프로젝트 개요
##### 대형 콘서트, 한정판 상품 오픈런 등 단기간에 폭발적인 트래픽이 몰리는 상황을 가정하여 서버 장애 없이 안정적으로 트래픽을 처리하고 100%의 데이터 정합성을 보장하는 선착순 예매 시스템 구축

# 사용 기술
+ ##### Java, Spring Boot, Spring Data JPA, Redis, MySQL

# 아키텍처
##### 1. Virtual Queue (가상 대기열) : Redis ZSet 을 활용하여 접속한 모든 유저에게 대시 순번을 부여하고 무한 로딩 대신 실시간 대기 상태 제공
##### 2. Scheduler Throttling (트래픽 제어) : Spring Scheduler 를 통해 DB가 감당할 수 있는 적정 수준의 인원만 결제 영역(Active Set) 으로 입장 허용
##### 3. Distributed Lock (분산 락) : 입장한 유저들 간의 동시성 문제는 Redisson 분산 락을 통해 제어하여 초과 예매 원천 차단

# 문제 해결 사례
#### 1. DB 커넥션 병목으로 인한 서버 장애 방어
+ ##### 문제: 2000명의 동시 접속 부하 테스트 시, 메인 DB의 커넥션 풀을 획득하기 위한 병목이 발생하여 ConnectionTimeoutException 이 쏟아지고 서버가 마비되는 현상 확인.
+ ##### 해결: DB 락 대신 메모리 기반의 빠른 Redis 분산 락으로 전환하고 그 앞단에 가상 대기열 시스템을 배치. DB로 향하는 직접적인 부하를 차단하고 대기열에서 트래픽을 통제함.
+ ##### 결과: DB 커넥션 타임아웃 에려율 0% 달성.
#### 2. 부하 테스트 임계점 도출 및 스케줄러 듀닝
+ ##### 문제: 대기열을 뚫고 결제 영역에 진입한 유저들이 분산 락 획득 대기 시간 10초을 버티지 목하고 예매 실패 후 튕겨 나가는 현상 발견
+ ##### 해결: 스케줄러가 한 번에 들여보내는 인원수를 극단적으로 조절해가며 부하 테스트 반복 수행. DB 처리 속도와 CPU 컨텍스트 스위칭 한계를 고려하여 1회 입장 허용 인원을 20명으로 최적화, 분산 락 획득 대기 시간을 60초로 변경.
+ ##### 결과: 대기열을 통과한 유저는 병목 없이 쾌적하게 결제를 완료할 수 있도록 시스템의 적정 TPS를 도출하고 UX를 개선

# 정략적 성과 요약
+ #### 동시성 환경 검증: CountDownLatch 와 ExecutorService 를 활용한 다중 스레드 통합 부하 테스트 코드 작성
+ #### 최종 성과: 
    ##### 1. 10,000명의 동시 접속과 10,000석 예매 시나리오에서 시스템 마비 및 튕김 현상 없이 10,000석 전량 안전하게 처리 완료
    ##### 2. 200,000명의 동시 접속과 16,000석 예매 시나리오에서 시스템 마비 및 튕김 현상 없이 16,000석 전량 안전하게 처리 완료
